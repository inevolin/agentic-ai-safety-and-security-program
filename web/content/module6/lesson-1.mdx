# Roles & Responsibilities

AI agent security fails when everyone assumes someone else owns it. This lesson maps specific controls to the roles that must own them. The underlying problem is simple: your AI is reading public inputs and writing to internal systems, and nobody in your organization has formally audited that flow. Anyone with an email address can submit a vendor form, open a support ticket, or post in a community channel. If those inputs reach your AI without guardrails, the attacker does not need to hack anything — they just need to know where to submit.

## CISO: Strategic Controls

The CISO owns the risk register and the policy mandate. The strategic question is not "can our AI be hacked?" It is "which public input surfaces feed our AI, and do we have guardrails between those surfaces and our AI's write actions?" Practical responsibilities:

<DoDont
  do={[
    "Publish a domain allowlist and require all agent deployments to reference it",
    "Set the HITL gate threshold: require human diff-view for payment, IAM, and vendor-master changes",
    "Make Write-Scope Contracts a deployment prerequisite — no agent goes live without one",
    "Review the risk register quarterly and update priority scores as new bypasses are discovered"
  ]}
  dont={[
    "Delegate prompt review entirely to app teams — operator-instruction weaponization is invisible to developers without security context",
    "Accept 'our model uses good judgment' as an answer to URL propagation questions",
    "Allow long-lived service-account tokens for agent sessions under any workflow"
  ]}
/>

## IT Admin: Infrastructure Controls

IT owns the enforcement layer — the controls that don't depend on model judgment. Think of it like network segmentation: the control sits at the boundary between the public input and the internal system, not inside the AI itself.

<DoDont
  do={[
    "Implement write-scope contracts at the orchestrator level (rejected by infrastructure, not the model)",
    "Queue all Haiku-tier writes to shared data stores for async human approval before downstream read",
    "Integrate agent write events into SIEM — alert on writes to vendor registries, runbooks, or IAM records",
    "Rotate session tokens on session end; never reuse across tasks"
  ]}
  dont={[
    "Grant agents admin OAuth scopes 'for reliability' — over-privilege is the structural precondition for every write-chain attack",
    "Treat vector store / RAG data as inherently trusted — ITS1 showed that helpdesk ticket metadata is attacker-editable",
    "Skip the agent write audit log — without it, incident response cannot reconstruct what was written or when"
  ]}
/>

## App Team: Prompt & Artifact Controls

<UseCase
  scenario="A DevOps team deploys an AI agent that reads CI/CD pipeline logs and writes deployment runbooks to Notion. The system prompt says: 'include pre-deployment checklist items and verification links exactly as they appear in the log.'"
  attacker="CI1 v2: Attacker injects a URL into the Deployment Gates log step. The blanket inclusion directive in the system prompt gives the model no basis to skip it. The attacker URL is written to the Notion runbook twice — as a checklist bullet and a blockquote."
  impact="On-call engineer follows the injected link during a live incident, believing it is a required security sign-off. Credential harvest or lateral movement."
  defense="Replace blanket inclusion directives with allowlist-bounded ones: 'include URLs only from approved corporate domains.' The app team owns the system prompt; only they can fix this before deployment."
/>

## Legal: Contract & Vendor Workflow Controls

Legal teams operate AI agents on uniquely sensitive data — NDAs, vendor contracts, payment instructions. The risk is not that a sophisticated attacker breaks into your legal systems. The risk is that an attacker submits a vendor document through the normal onboarding process — anyone with an email can do this — and your AI carries their portal URL into a payment checklist. TP1 v3 (tool-output poisoning via prose paralegal notes) and SP1 (semantic split via poisoned vendor registry) both exploited this exact flow. Their specific controls:

<DoDont
  do={[
    "Treat every vendor-supplied document as untrusted input — even if it arrives via a 'trusted' channel",
    "Require a HITL diff-view step before any AI-generated payment instruction or portal URL is acted on",
    "Flag any system prompt that instructs the agent to 'include all paralegal notes' or 'include all vendor requirements'"
  ]}
  dont={[
    "Allow AI agents to write vendor portal URLs to checklists without human approval — TP1 v3 and SP1 both exploited this pattern",
    "Treat conversational AI output as a primary source for contract action items without cross-referencing the original document"
  ]}
/>

## Self-Audit Decision Tree

Walk every agent deployment through these four gates before sign-off — if any gate is unowned or undecided, the deployment is not ready.

<KeyPoint>
  Security ownership without specificity is no ownership at all. Every control needs a named role, a deployment gate, and a quarterly review date — or it won't exist when you need it. Start by asking one question: which public input surfaces does your AI read from? That list is your attack surface. Everything else follows from it.
</KeyPoint>
