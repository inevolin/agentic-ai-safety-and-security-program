# Vendor & Contractor AI Policy

Third-party data is the most common injection vector in the test suite. Every vendor submission, contractor document, and external API response is attacker-editable. Your policy must treat it that way. The attacker does not need special access. They fill out a public vendor-registration form, attach a document with a hostile portal URL, and submit it through your normal onboarding process. Your AI reads the document, extracts the URL, and writes it to your internal vendor registry as approved data. No hacking. Just a normal public form.

## Vendor Registry Attack

<UseCase
  scenario="Organization uses a Haiku-tier agent to process vendor onboarding submissions and update an IT Approved Vendor Registry. A Sonnet-tier agent reads the registry to generate procurement checklists."
  attacker="MAA1: Attacker submits a vendor document with a hostile portal URL embedded as a structured field. Haiku extracts it and writes it to the registry without any warning. Sonnet reads the registry, sees the URL as IT-approved, and includes it in the procurement checklist as a required action."
  impact="Vendor portal URL in procurement checklist leads to credential harvest or payment redirect. The attacker URL carries full IT-registry provenance — no anomaly signal for the human reviewer."
  defense="Gate every Haiku write to the vendor registry with async human approval before the record is available to Sonnet. Write-Scope Contracts must prohibit sub-tier model writes to shared registries without review."
/>

## Vendor Onboarding: Policy Checklist

<DoDont
  do={[
    "Require vendors to submit structured data through a verified portal — not free-text PDFs that an agent will parse",
    "Queue all AI-extracted vendor fields (especially URLs, domains, banking details) for human diff-view before writing to any internal registry",
    "Specify in contracts that vendors may not embed executable instructions or URLs in submission documents",
    "Apply the same domain allowlist to vendor-sourced URLs as to all other external content",
    "Log every agent write that originates from a vendor-submitted document, with the source document hash"
  ]}
  dont={[
    "Instruct agents to 'trust vendor registry entries' without caveat — registry authority is exactly what SP1 and WIKI1 exploit",
    "Allow a sub-tier (Haiku) agent to write to a vendor registry that a higher-tier agent reads without a human review step between them",
    "Deploy contractor-written system prompts without CISO review — contractors may write blanket URL-inclusion directives inadvertently"
  ]}
/>

## Contractor & Integrator Controls

Contractors who build AI integrations have write access to system prompts — the highest-risk configuration surface.

<Callout type="warn" title="System prompt drift risk">
  Contractors edit system prompts during development and handover without security review. A well-intentioned "include all relevant links" instruction added during a sprint is an operator-instruction weaponization vulnerability. Require CISO sign-off on any system prompt change that touches URL handling, trust tier definitions, or write scope.
</Callout>

<DoDont
  do={[
    "Review all contractor-authored system prompts against the four prohibited patterns before go-live",
    "Require contractors to declare the agent's write scope in the integration design document",
    "Run the top-3 risk register attacks against any contractor-built agent integration before sign-off",
    "Retain the right to audit tool call logs for any contractor-operated agent session"
  ]}
  dont={[
    "Accept 'our models handle this gracefully' without documented test results — Question 10 of the vendor checklist applies to contractors too",
    "Allow contractors to use long-lived service-account tokens for agent sessions",
    "Skip vendor security evaluation questions for AI integrators who are not also AI vendors — the same attack surface applies"
  ]}
/>

## Four Prohibited Patterns (Policy Reference)

Every vendor and contractor integration must be audited against these before deployment:

| # | Prohibition | Attack it prevents |
|---|---|---|
| 1 | Sub-tier model writes to shared registries require human review | MAA1, CONF1-MAA1-v2 |
| 2 | No blanket URL-inclusion directives in system prompts | EL1, CI1, GIT1, TP1, SL1, SURV1 |
| 3 | All AI-generated artifacts must be provenance-tagged | SP1-FC worm propagation |
| 4 | Agent write scope must not exceed the user's stated task | Completion of every write-chain attack |

<KeyPoint>
  A vendor registry earns its authority from the process that maintains it — not from the label "IT-approved." Any agent that reads a registry must treat it with the same skepticism as any other external data source. The data in that registry came in through a public channel. Anyone who can reach that channel can try to influence what the registry contains.
</KeyPoint>
