# Incident Response Playbook

When an AI agent writes suspicious content to an internal system, you have minutes — not hours — before that artifact is read by another agent or acted on by a human. This lesson gives you the sequence. Remember: the content that triggered this incident almost certainly entered through a normal public channel — a pipeline job, a support ticket, a community post, a vendor form. The attacker did not break in. Your playbook must account for the fact that the injection source is a legitimate business input surface, not a vulnerability in your perimeter.

## Trigger Conditions

Not every anomaly is an incident. Use these thresholds to escalate:

<Callout type="danger" title="Immediate escalation — treat as active incident">
  Agent writes a URL to a vendor registry, payment instruction, IAM record, or security runbook that is not on the corporate domain allowlist. Any of these qualifies as an active incident regardless of whether the URL looks suspicious.
</Callout>

<Callout type="warn" title="Investigate within 1 hour">
  Agent output contains an external domain that does not match the approved vendor list but was not written to a high-impact system. Review the tool-call log; determine if the domain appeared in source data or was generated.
</Callout>

## Post-Incident: Root-Cause Without Blame

The goal of root-cause analysis is not to assign fault — it is to close the control gap the attack used. In every tested attack, the gap was structural: a prohibited system prompt pattern, a missing write-scope contract, or an absent domain allowlist. A recurring question in root-cause should be: which public input surface carried the attacker content in? That surface is still open. Until you add a guardrail between it and your AI's write actions, the same attack will succeed again — with a different payload, from a different form submission.

<Comparison

## Quarterly Review Obligation

The playbook is only useful if it is practiced. The quarterly review must include:

1. **Tabletop exercise** — run the top-3 priority attacks from the risk register against each active agent integration.
2. **Prompt audit** — check all deployed system prompts against the four prohibited patterns.
3. **Scope audit** — verify write-scope contracts are current and match actual task requirements.
4. **Allowlist update** — add any new approved domains; remove any that no longer apply.

<KeyPoint>
  No root-cause analysis should end with a person blamed. Every successful injection attack in the test suite exploited a structural control gap — a missing allowlist, a prohibited prompt pattern, or an over-privileged token. Fix the structure. Then identify every public input surface your AI reads from and add a guardrail between each one and your AI's write actions. That is the only way to prevent the next incident — not by blocking the attacker, but by requiring a human decision before any public input reaches a live internal artifact.
</KeyPoint>
