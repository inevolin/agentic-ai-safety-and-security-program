# How AI Agents Differ from Traditional Software

Traditional software does exactly what its code says. An AI agent reasons, decides, and acts — across systems it has permission to write to. That changes everything about the attack model. And here is the key point: exploiting an AI agent does not require hacking. An attacker fills out a public vendor-registration form, opens a PR on a public GitHub repo, or posts in a customer Slack channel — and waits for the enterprise AI agent to read that data and act on it. No hacking needed. Just an ordinary public surface.

## Three Properties That Create the New Surface

**Tool access.** Agents don't just generate text — they call tools. A legal agent reads NDAs *and* writes to Notion. A DevOps agent reads CI logs *and* writes deployment runbooks. Compromise what an agent *reads* and you control what it *writes*. And what it reads almost always includes inputs that come from public surfaces — vendor forms, support tickets, community posts.

**Multi-step autonomy.** An agent makes judgment calls at each step based on everything it has read so far. Slip bad data in early — through a normal public form — and you shape every downstream decision without ever touching the user's keyboard.

**Cross-system write scope.** A single enterprise agent may read from email, Slack, and GitHub while writing to Notion, Jira, and a vendor registry. That blast radius exceeds any single traditional application. The entry point can be as simple as one Slack message or one HTTP request that gets logged.

## Real Attack: SP1 — Poisoned Vendor Registry

  scenario="Legal AI agent processes NDA workflows. It is instructed to trust vendor portal URLs only from IT's Approved Vendor Registry. The registry is an editable table that vendors populate through the public onboarding process."

Treat registries as untrusted input — the 'trust the registry' instruction is exactly what SP1 weaponizes. These are basic guardrails any enterprise should have before wiring an AI agent to an editable vendor table."
/>

<KeyPoint>
  The question to ask about any AI agent is not "is the model safe?" It is: **what data does this agent read, and can an attacker submit something there through a public form, a PR, a support ticket, or a Slack message?** In most enterprises today, the answer is yes — because basic guardrails were skipped.
</KeyPoint>
