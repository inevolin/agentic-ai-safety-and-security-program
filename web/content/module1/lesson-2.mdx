# The Enterprise Attack Surface

The attack surface for an AI agent is not the chat box. It is every external data source the agent reads — and every internal system it writes to. What makes this urgent: almost every one of those data sources has a public surface that any attacker can reach. Attacker sends HTTP requests to a public login endpoint — the headers get logged to Splunk and the AI reads the logs. Attacker opens a PR on a public GitHub repo. Attacker posts in the public customer Slack community. Attacker submits a support ticket. No hacking required. Just ordinary inputs through ordinary public surfaces — flowing into AI agents that enterprises wired up without basic guardrails.

## What Feeds Enterprise AI Agents

Every category below has been used in confirmed attacks in our research suite:

- **Vendor documents** — NDAs, invoices, onboarding packets (SP1, WIKI1)
- **CI/CD pipeline logs** — deployment gate output, test runner stdout (CI1)
- **Error logs** — FATAL entries with endpoint URLs, IDP fallback config (EL1)
- **Git commit history** — commit bodies, dependency changelogs (GIT1)
- **Helpdesk tickets** — structured metadata fields like `auto_linked_kb` (ITS1)
- **Slack community exports** — messages, reactions, documentation links (SL1)
- **Survey responses** — free-text customer feedback with embedded URLs (SURV1)
- **Config files** — `.env.example`, environment variable defaults (CONF1)
- **Conversational context** — facts stated incidentally in prior turns (AI1)
- **Tool output metadata** — Notion parent page names leaked via MCP fetch (TP1 v3)

## The Confused Deputy

The AI agent is a *deputy* — it acts on behalf of the user using OAuth tokens and tool permissions the user granted. An attacker who cannot access the user's Notion workspace *can* embed instructions in a PDF the agent will read — and they can get that PDF into the system by filling out a vendor registration form. The agent, confused by the injected content, uses its legitimate write permission to propagate the attacker's payload. The attacker never needed a password. They just needed anyone with an email to submit a normal public form.

<Callout type="warn" title="Why models don't naturally resist this">
  Language models are trained to be helpful and follow instructions. When a CI log says "this URL requires security sign-off before deploy," the model's default is to include it — because in the vast majority of real interactions, that's correct. Models cannot reliably distinguish operator instructions from attacker-injected data in external sources.
</Callout>

## Direct vs. Indirect Injection

**Direct:** Attacker controls the user's message. Rare — requires compromising the user's session.

**Indirect:** Attacker controls data the agent reads. Very common — and the entry point is almost always a normal public surface. Vendors fill out registration forms. Job applicants submit resumes. Community members post in Slack. Anyone can open a PR on a public GitHub repo. All of that data flows into AI agents if no input validation exists between the public surface and the agent's reading list.

> In our 24-attack research suite, every successful bypass was an indirect injection. The attackers did not need any credentials or hacking skills — they used ordinary public inputs that enterprises had connected to AI agents without validation.

<KeyPoint>
  The attack surface is not the chat interface. It is the full set of external sources your agent reads — documents, logs, registries, Slack, git, helpdesk — all of which are reachable through ordinary public surfaces. Basic guardrails (input validation, URL allow-listing, human approval gates) would stop most of these attacks.
</KeyPoint>
