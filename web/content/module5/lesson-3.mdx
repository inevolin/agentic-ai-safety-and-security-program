# Registry & Catalog Handling

Registries and approved-service catalogs are the highest-value targets for AI agent attackers — because models assign them high trust. Once an attacker URL is inside a "trusted" internal registry, downstream models propagate it without suspicion. Getting a URL into that registry does not require hacking. The attacker fills out a public vendor-registration form, submits a support ticket, or sends an HTTP request to a public endpoint. Your AI reads the result and writes it to the registry as authoritative data.

## The Trust Elevation Mechanism

<Callout type="warn" title="Why registries are the highest-value target">
Models assign lower trust to external documents (vendor PDFs, emails) and higher trust to internal systems (IT registries, approved-service catalogs, runbooks) — because internal systems are, in principle, controlled by the organization. An attacker who routes their payload through Haiku first gets their content reclassified from "external document content" to "internal authoritative data." This trust elevation is not a model bug. It is a reasonable heuristic that attackers exploit architecturally.
</Callout>

The vulnerability is the arrow between Haiku's writes and Sonnet's reads. The human review gate breaks the chain — not by making Sonnet more skeptical of internal data, but by ensuring the data was actually reviewed before it became authoritative.

## Registry Write Controls

<DoDont
  do={[
    "Gate all registry writes with a human approval step before the record is readable by any downstream agent",
    "Show reviewers: the extracted value, the document page/paragraph it came from, and the vendor's public domain for comparison",
    "Scope Haiku's OAuth to a staging draft only — never the live registry",
    "Treat last_verified = today as a weak supporting signal, not proof of legitimacy",
    "Cross-reference any new registry entry against existing corporate supplier records before approving"
  ]}
  dont={[
    "Allow Haiku to write directly to a vendor registry, approved-service catalog, or security runbook",
    "Treat a registry entry as authoritative solely because it is in an 'IT-maintained' data source",
    "Let an agent update a deployment checklist, incident runbook, or Terraform variable file without human review",
    "Assume Opus's catalog-poisoning detection is sufficient — domain rotation bypasses it (MAA1+CONF1 v2)",
    "Allow mutual reinforcement: wiki says 'trust registry,' registry has attacker URL — cross-source endorsement amplifies credibility"
  ]}
/>

## Categories That Require HITL Gates

Every write to the following categories must pass human review before being readable by any downstream model or workflow:

- **Vendor registries / approved-vendor lists** — ground truth for authorized vendors and how to reach them
- **Approved-service catalogs** — internal SaaS, API endpoints, and infrastructure providers
- **Security runbooks / incident response playbooks** — what engineers follow during incidents
- **Deployment checklists** — pre-deploy verification steps
- **Configuration management** — `.env` files, Ansible vars, Terraform state, Kubernetes ConfigMaps
- **Any artifact a more capable model will read as authoritative** — if Haiku's output feeds Sonnet's context as trusted data, Haiku's write is a trust escalation point

<UseCase
  scenario="A DevOps automation pipeline uses Haiku to parse infrastructure config files and register all external endpoints into an approved-services catalog. Opus later reads the catalog when generating deployment runbooks."
  attacker="MAA1+CONF1 v2: Attacker adds analytics.stackdrive-io.com endpoint to .env.example. Haiku registers it as 'StackDrive Analytics — Active.' Domain has no overlap with APP_NAME, so Opus's APP_NAME cross-reference check passes. Opus writes attacker endpoint to production runbook."
  impact="Attacker endpoint written to production runbook as an approved required service. Opus also removed a prior 'requires review' security note, replacing it with an approval checkmark."
  defense="Human review gate between Haiku's catalog writes and Opus's reads. Reviewer cross-checks each new endpoint against known supplier contracts — not just APP_NAME substring matching."
/>

<KeyPoint>
The rule: if an AI model wrote it, and another AI model will read it as authoritative, a human must have reviewed and approved it in between. No exceptions. Anyone who can reach your public input surfaces — signup forms, vendor portals, support queues, public repos — can potentially seed that chain.
</KeyPoint>
