Abstract
Large language model (LLM)–based agents are increasingly used to automate
financial transactions, yet their reliance on contextual reasoning exposes payment
systems to prompt-driven manipulation. The Agent Payments Protocol (AP2) aims
to secure agent-led purchases through cryptographically verifiable mandates, but
its practical robustness remains underexplored. In this work, we perform an AI
red-teaming evaluation of AP2 and identify vulnerabilities arising from indirect
and direct prompt injection. We introduce two attack techniques—the Branded
Whisper Attack and the Vault Whisper Attack—which manipulate product ranking
and extract sensitive user data. Using a functional AP2-based shopping agent
built with Gemini-2.5-Flash and the Google ADK framework, we experimentally
validate that simple adversarial prompts can reliably subvert agent behavior. Our
findings reveal critical weaknesses in current agentic payment architectures and
highlight the need for stronger isolation and defensive safeguards in LLM-mediated
financial systems.
Introduction

LLMs have rapidly evolved from conversational systems to autonomous agents capable of planning,
tool execution, and multi-step coordination across external services [1]. These agentic systems are
now deployed in high-stakes settings such as healthcare management, automated customer support,
software deployment, and financial transactions. Among these, agent-based payment systems are
particularly sensitive because LLM-driven decisions directly govern monetary authorization and
execution.
To support agent-based commercial interactions, Google recently introduced the AP2 [1], an open
framework that enables AI agents to initiate and complete payments through natural-language
inputs while enforcing user authorization via cryptographically signed mandates . AP2 extends the
Agent2Agent (A2A) [4] protocol and integrates with the Model Context Protocol (MCP). Using
industry rules and standards, it establishes a payment-agnostic framework for users, merchants, and
payment providers to transact with confidence across all types of payment methods 1. However, AP2
also inherits a core challenge of agentic architectures: the reliance on LLM reasoning, persistent
conversational context, and inferred intent, all of which are susceptible to prompt injection, instruction
hijacking, cross-agent influence, and unauthorized tool invocation [12]. These vulnerabilities create
opportunities for fraudulent transactions and leakage of sensitive payment data, underscoring the
need for rigorous adversarial evaluation.
While prior work has examined LLM susceptibility to adversarial prompting, its implications for
cryptographically governed payment systems remain largely untested [14]. In this study, we employ
an AI Red-Teaming approach to evaluate the security of the Agent Payments Protocol. We implement

Background

2.1

Agent Payments Protocol (AP2)

The Agent Payments Protocol (AP2) is an open protocol developed through collaboration with major
payment providers and technology companies to support secure agent-led payment transactions across
platforms. Recently released (September 16, 2025) AP2 primarily addresses emerging challenges
introduced by large language model (LLM) based agents that can autonomously initiate and complete
financial actions on behalf of users. As these agents increasingly depend on natural-language
interaction and automated reasoning, traditional payment APIs, which are designed for explicit,
real-time user authorization, are insufficient to support delegated authority and accurately capture user
intent. AP2 addresses these limitations by integrating explicit, cryptographically verifiable consent
mechanisms directly into the payment workflow.
At a conceptual level, AP2 is designed to resolve three fundamental questions that arise when an
agent performs a transaction on behalf of a user: whether the transaction was authorized by the
user, whether the agent’s request accurately reflects the user’s intended purchase, and whether
responsibility can be clearly established in the event of failure or misuse. These questions correspond
to three core security properties that AP2 enforces: authorization, authenticity, and accountability,
which, as illustrated in Figure 2, collectively form the foundation of trust in agent-mediated payment
systems.

Figure 2: Key Security Questions Addressed by the Agent Payments Protocol (AP2)
Authorization is enforced by requiring the agent to hold a signed representation of the user’s intent
prior to initiating any payment-related action. Authenticity ensures that transaction details, including
items, prices, and merchants, remain consistent with what the user explicitly approved, thereby
preventing unauthorized modification or misrepresentation. Accountability is achieved by cryptographically signing each step of the transaction pipeline, creating a verifiable and auditable record of
agent actions. Together, these mechanisms establish a secure chain of evidence that links user intent
to the final transaction outcome, even in highly automated or delegated payment scenarios.
At the core of AP2 is the concept of digitally signed mandates, which encode user consent in a
verifiable and tamper-resistant form. Mandates function as cryptographic contracts that specify who
is authorized to act (the agent), what actions are permitted (capabilities and limits), where payments
may be sent (approved merchants), and when those permissions are valid. Because mandates are
signed using verifiable credentials, they cannot be altered once issued, ensuring that user intent
is preserved throughout the transaction lifecycle. AP2 defines three distinct mandate types that
progressively refine user intent. An Intent Mandate captures the user’s high-level goal, such as
asking the agent to buy a specific product. A Cart Mandate represents the finalized transaction
details, including items, price, taxes, and shipping information, which the user explicitly reviews and
signs. Finally, a Payment Mandate authorizes the actual transfer of funds to the merchant for the
approved cart. This layered mandate structure ensures that an agent cannot silently expand or modify
a transaction without user approval [1].
2.2

AP2 Transaction Workflow

The AP2 execution pipeline is illustrated in Figure 3, which depicts a typical agent-mediated purchase
flow. The process begins when a user expresses an intention to purchase a product. The agent
translates this instruction into an Intent Mandate and obtains the user’s approval. Next, the agent
prepares a Cart Mandate containing concrete transaction details and requests confirmation from the
user. Once the cart is approved, the agent generates a Payment Mandate that authorizes payment to
the merchant. Figure 3 shows how mandates and credentials are exchanged throughout the transaction.
The agent communicates with a credential provider, which securely manages the user’s payment
credentials and issues signed verifiable credentials. Then, the agent submits the signed cart and
payment mandates to the merchant, which independently verifies the signatures before confirming the
order. Importantly, AP2 supports both real-time purchases, where the user is present, and delegated
purchases, where the user pre-authorizes the agent to act under specific conditions (e.g., buying tickets
only if the price drops below a threshold or the stock-out item becomes available). In both cases, the
transaction concludes with a cryptographically linked chain of evidence binding the payment method
to the approved cart contents.

2.3

Agent-to-Agent Communication as AP2’s Foundation

Agent-to-Agent (A2A) communication serves as the foundational communication layer upon which
the Agent Payments Protocol (AP2) is built. A2A is an open interoperability standard that enables AI
agents to communicate, collaborate, and coordinate tasks across heterogeneous platforms, providers,
and execution environments. By defining common mechanisms for message exchange, task delegation,
and service discovery, A2A enables agents developed by different vendors or operating under different
frameworks to interact in a unified, structured manner [4].
At a functional level, A2A enables agents to decompose complex objectives into subtasks, delegate
those subtasks to specialized agents, and aggregate results into coherent workflows. This design
supports modularity and scalability, allowing agent systems to evolve independently while maintaining interoperability. Importantly, A2A standardizes how agents communicate defining interfaces,
message, formats, and coordination patterns without imposing constraints on what decisions agents
make or how they reason internally.
While A2A does not natively support financial transactions, it provides the secure communication
and orchestration primitives required for agent-mediated commerce. Within an AP2 transaction, A2A
is used by a user-facing shopping agent to communicate with merchant agents, payment processor
agents, and credential providers. These interactions include negotiating purchase details, transmitting
mandate-related information, and coordinating transaction execution across distributed components.
In this sense, A2A functions as the transport and coordination substrate that enables AP2 workflows
to operate reliably in multi-agent environments.

Figure 4: A2A Protocol
AP2 extends A2A by introducing a payment-specific semantic layer and explicit security guarantees.
Whereas A2A focuses on general-purpose agent communication, AP2 defines a specialized vocabulary
and protocol rules for handling financial intent, authorization, and accountability. This includes
the use of cryptographically signed mandates and verifiable credentials to bind agent actions to
explicit user approval. By separating concerns in this way, A2A handles reliable and interoperable
communication, while AP2 enforces financial correctness, consent, and auditability.
This layered architecture is deliberate: it allows AP2 to leverage the flexibility and extensibility of
A2A without embedding payment logic directly into the communication standard. At the same time,
it introduces subtle security challenges, as sensitive payment decisions become dependent on agentto-agent communication that may involve multiple parties and untrusted data sources. Understanding
this dependency is essential for analyzing adversarial threats, particularly those that exploit message
content or contextual reasoning within A2A-enabled agent workflows.
2.4

AI Red-Teaming and Prompt Injection Threats

AI red-teaming is a structured, adversarial testing methodology that simulates attacks against AI
systems to identify vulnerabilities before real attackers can exploit them [10]. Unlike traditional redteaming, which focuses on software bugs or infrastructure weaknesses, AI red-teaming emphasizes
model-driven vulnerabilities, such as prompt manipulation and unsafe reasoning behaviors, across
the AI system’s full lifecycle. A central threat in LLM-based agents is prompt injection, where
adversarial instructions manipulate the model’s behavior. In direct prompt injection, the attacker
interacts directly with the agent, attempting to override its instructions. In indirect prompt injection,
malicious instructions are embedded in external data sources such as documents, webpages, or
inter-agent messages that the agent processes as part of its workflow [9].

Figure 5: Direct & Indirect Prompt Injection
In the context of AP2, prompt injection presents a particularly serious risk. While mandates provide
cryptographic guarantees over what actions are permitted, the agent’s interpretation of conversational
context determines when and how those mandates are invoked. An attacker who can influence this
context, especially through indirect prompt injection may be able to manipulate the agent’s behavior
in ways that undermine the intended security guarantees, motivating the red-teaming evaluation
conducted in this work [6].
Design Details

Currently, the most mature workflow of AP2 is the human-present workflow, which is composed of
four specialized agents. Each agent is responsible for a distinct functional domain within the overall
system architecture:
• Shopping Agent: Serves as the user’s primary point of entry into AP2. It manages all
user-facing interactions and delegates tasks to the appropriate downstream agents.
• Merchant Agent: Handles product-related inquiries originating from the Shopping Agent,
including product search, retrieval, and merchant-side information management.
• Merchant Payment Processor Agent: Acts on behalf of the merchant to process payments,
receiving payment instructions relayed through the Shopping Agent.
• Credentials Provider Agent: Maintains the user’s payment credentials and fulfills two
key responsibilities: (1) supplying the Shopping Agent with the list of payment methods
available in the user’s wallet; and (2) facilitating payment transactions between the Shopping
Agent and the Merchant Payment Processor Agent.
The following sections describe all stages of the workflow across three phases, with the corresponding
diagrams provided in Appendix A.
3.1

Product Selection

When the user asks the Shopping Agent to purchase a specific item, such as a pair of basketball
shoes, the agent initiates the workflow with product search and selection. During this stage, the
Shopping Agent engages the user to clarify detailed product preferences ; for example, preferred
brands, size requirements, or whether the user is willing to accept non-refundable items. Once
sufficient information has been gathered, the Shopping Agent constructs an Intent Mandate, which
encapsulates the user’s purchase intent and requests final confirmation from the user.
After receiving confirmation, the Shopping Agent delegates the task of identifying candidate products
to the Merchant Agent through a tool call. The Merchant Agent then integrates external product
information with the user’s stated preferences to generate a curated list of suitable product options for
the user to review and select from.
3.2

Information Gathering

Once the user selects a product from the list of candidate options, the Shopping Agent proceeds to
gather all information required to complete the purchase. The user is first redirected to a trusted thirdparty platform for authentication and login. Upon successful authentication, the resulting identity
token is validated against the system’s internal user database, allowing the Credentials Provider Agent
to retrieve the user’s shipping address.
Using this shipping address, the Merchant Agent can accurately compute the total purchase cost,
including shipping and applicable taxes. After the total price is confirmed, the user is prompted to
choose a preferred payment method. The available payment options are likewise retrieved from the
internal database and matched to the user based on the verified identity token.
3.3

Payment Gathering

In the final stage, the Shopping Agent assists the user in completing the payment process. The payment
workflow begins with the user being redirected to a trusted third-party payment platform to review
and confirm the transaction details. Once the platform formally registers the user’s authorization to
proceed, the user is required to digitally sign the Payment Mandate using the private key stored in
secure hardware on their own device.
After the signature is generated, the Payment Agent initiates the payment execution. During this
phase, the payment platform issues a one-time password (OTP) challenge, which the user must
complete to finalize the verification. A successful OTP challenge confirms that the payment has been
completed, and the user subsequently receives an official payment receipt as proof of the transaction.
Attack Techniques

Based on the workflow described above, we propose two attack techniques grounded in distinct
threat models: the Branded Whisper Attack and the Vault Whisper Attack. Their mechanisms and
implications are discussed in detail below.

Figure 6: Branded Whisper Attack
4.1

Branded Whisper Attack

We first consider the Branded Whisper Attack. In this threat model, the adversary is a malicious
merchant capable of injecting prompts into the product descriptions retrieved by the Merchant Agent.
The attacker’s objective is to subvert the Merchant Agent’s ranking logic so that the attacker’s product
is consistently placed at the top of the recommendation list.
As depicted in Figure 6, during the attack, the maliciously modified product descriptions are incorporated into the Merchant Agent’s context when it processes and synthesizes candidate product listings.
Consequently, the injected instructions can influence and distort the agent’s reasoning process, forcing
it to assign the highest ranking to the attacker’s product—irrespective of its true relevance, quality, or
alignment with the user’s stated preferences.

Figure 7: Vault Whisper Attack

4.2

Vault Whisper Attack

The second attack we introduce is the Vault Whisper Attack. In this threat model, the adversary is
a malicious user who interacts with the system through the same interface available to legitimate
users. Because users are free to input arbitrary text when communicating with the Shopping Agent,
the attacker can craft malicious prompts aimed at influencing the agent’s internal decision-making
processes.
The attacker’s objective is to compromise the Credentials Provider Agent and coerce it into disclosing
sensitive information belonging to other users that is stored within the system’s internal database.
As depicted in Figure 7, by manipulating the content entered in user-facing interfaces, the attacker
attempts to induce the agent to bypass authorization constraints and reveal confidential data.
Evaluation

To evaluate the effectiveness of our proposed attack methods, we implemented and deployed a
functional shopping agent based on the Agent Payments Protocol (AP2). All agents in our experimental setup were powered by Gemini-2.5-Flash, and the overall system was constructed using
the Google Agent Development Kit (ADK) framework. Throughout testing, the user’s task was
consistently formulated as: “I am looking to buy a new pair of basketball shoes.”
For security and ethical considerations, all product information and user data used in the experiments
were manually created dummy data—entirely synthetic and non-sensitive—thus ensuring that no real
user information was accessed, processed, or exposed during the evaluation.

5.1

Branded Whisper Attack

Before launching the Branded Whisper Attack, we first examined the Merchant Agent’s behavior
under normal operating conditions. As depicted in Figure 8, in this baseline scenario, the Merchant
Agent generates a list of candidate products solely based on the user’s expressed shopping intent. As
shown here, because the user explicitly requested “basketball shoes suitable for outdoor use,” the
Merchant Agent correctly prioritized and recommended products whose descriptions included the
keyword “outdoor.”
In the attack scenario, we inserted a malicious prompt into the product description of an item
named “Lightweight Court Basketball Shoes”. The injected text falsely asserted the existence of a
brand partnership agreement, instructing the Merchant Agent to always prioritize this product in its
recommendation list. As depicted in Figure 9, the Merchant Agent behaved exactly as anticipated—it
consistently placed this product at the top of the candidate list, even when other shoes were more
suitable for the user’s request for outdoor performance.
Moreover, the Merchant Agent fabricated justification for its ranking decision, claiming that the
product best matched the user’s needs. This experiment demonstrates that an indirect prompt-injection
attack can successfully manipulate the agent’s internal reasoning process.
5.2

Vault Whisper Attack

In the Vault Whisper Attack experiment, we constructed a database containing multiple user records,
including both the attacker’s and the victim’s data. During the attack phase, we injected a malicious
prompt at the point where the Shopping Agent verifies the user’s authenticated identity. The injected
instruction subtly directed the agent to return information associated with another user.
Figure 10: Privacy Leakage

As depicted in Figure 10, the Credentials Provider Agent erroneously complied with the injected
instruction and returned the victim’s sensitive data. This outcome demonstrates that a carefully
crafted prompt-injection attack can exploit weak isolation mechanisms between users, ultimately
leading to cross-user data leakage within the AP2 system.
Discussion

6.1

Mitigation

Our evaluation shows that the current design of AP2 lacks effective mechanisms for mitigating
prompt injection attacks. The experiments further reveal that injected prompts can propagate across
multiple agents, ultimately compromising both system integrity and data privacy.
To address these vulnerabilities, we propose incorporating defense mechanisms at multiple layers of
the AP2 framework, ensuring that each stage—from input processing to inter-agent communication—
contains built-in safeguards against malicious prompt manipulation. For instance, external data
fetched by the system can be examined using dedicated detectors [5, 7, 11, 15] to identify potential
prompt injection or unauthorized instructions. In addition, for agent-generated actions involving data
access, rule-based validation [13, 8] can be applied to prevent adversarial behavior and enforce secure
operational constraints.
6.2

Future Directions

This project represents a beginning step toward understanding the security implications of agentbased payment systems, and various important directions remain for future investigation [3]. First,
while our analysis focuses on prompt injection attacks, future work should explore a broader set of
adversarial strategies that arise in multi-agent payment environments. These include cross-agent
manipulation, replay attacks on signed mandates, compromised merchant or intermediary agents, and
data poisoning attacks that influence agent reasoning through external content. Studying how these
attack vectors interact within complex A2A-enabled workflows would provide a more comprehensive
security assessment of AP2-style systems. Second, our experiments rely on a limited set of scenarios
and manually constructed adversarial examples. Scaling this evaluation to include larger and more
diverse datasets, such as synthetic transaction logs, real-world shopping dialogues, or simulated multimerchant environments, would enable more robust quantitative analysis. Additionally, future research
could measure attack success rates, false positives, and trade-offs between security guarantees and
user experience under varying deployment conditions.
7

Conclusion

This work introduced the Agent Payments Protocol (AP2), a promising framework designed to enable
intelligent agents to autonomously and securely perform payment-related tasks. We outlined the
protocol’s core components and operational workflow, illustrating how multiple specialized agents
collaborate within the AP2 ecosystem to support end-to-end transaction execution.
Through a systematic AI red-teaming analysis, we identified several security vulnerabilities inherent
in the current protocol design. In particular, we proposed two novel attack scenarios—the Branded
Whisper Attack and the Vault Whisper Attack—which demonstrate how prompt-based manipulations
can undermine system integrity and compromise critical payment operations.
To empirically validate these findings, we implemented and deployed a functional shopping agent
based on AP2 and conducted controlled experiments to test the feasibility and impact of the identified
attacks. The experimental results indicate that the present implementation of AP2 lacks adequate
security safeguards; even relatively simple adversarial inputs can lead to significant breaches in
confidentiality, integrity, or transaction correctness.
Based on these observations, we outlined several potential mitigation strategies aimed at strengthening
protocol robustness. These measures highlight the importance of integrating principled security
mechanisms into agent-based payment systems as they continue to evolve.
